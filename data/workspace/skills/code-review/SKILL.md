# Code Review

## Когда использовать
- Перед любым commit, PR или отчётом о завершённой кодовой задаче.
- После изменения TypeScript, тестов, tools, Telegram-команд или конфигурации.
- Когда проверка упала и нужно понять, чинить ли дальше.

## Workflow
1. Прочитай изменённые файлы, которые сам менял.
2. Проверь `git diff --stat` и `git diff` только по своим файлам.
3. Запусти строго по порядку: `npm run typecheck`, `npm run lint`, `npm test`, `npm run build`.
4. Если проверка упала, исправь минимально возможным изменением и повтори весь набор.
5. Делай максимум 3 попытки фикса на один failing файл или один класс ошибки.
6. Проверь, что тесты покрывают новый модуль или новый публичный путь.
7. Сформулируй результат без преувеличений: что изменено, что проверено, что осталось рискованным.

## Выходной формат
В Telegram напиши:

- changed: список файлов
- checks: `typecheck`, `lint`, `test`, `build`
- risk: низкий/средний/высокий и почему
- commit_ready: yes/no

## Стоп-условия
- После 3 попыток одна и та же проверка остаётся красной: остановись и напиши команду, файл и краткую причину.
- Неясно, чьи изменения в diff: не продолжай, попроси owner подтвердить scope.
- Нет теста для нового модуля: остановись или добавь минимальный тест.
